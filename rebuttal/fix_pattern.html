We classified fix completion into three groups according to whether they can be automatically achieved, shown in the table below.
  • Automatic: 28 fix patterns can be automatically realized without any human intervention. These fix patterns often involve the simple addition of existing libraries or classes or the removal of code.
  • Semi-Automatic: 13 fix patterns can only be semi-automatically achieved and need human intervention. These fix patterns often involve the addition of new method or class implementation that is related to business logic or the addition of variables that have many possibilities. Therefore, developers need to be involved in the fix process.
  • Blackbox: 7 fix patterns for syntax violation errors are difficult to achieve automatic fix. However, blackbox techniques like deep learning techniques have goog potential, as demonstrated by Santos et al. 2018 [42].

    • It is worth mentioning that, while automatic fixing for each specific fix pattern is feasible, it is difficult to determine which fix pattern should be used for a compiler error, i.e., the root cause analysis problem. We believe that automatic root cause analysis can be achieved by investigating the commit history: first identify the code changes in previous commits, then understand the various semantics underlying code changes (e.g., various refactorings) and determine the impact of code changes, and finally establish the cause-effect links between code changes and compiler errors.
